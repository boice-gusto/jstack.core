# jstack markdown system

This document maps how prose and machine-facing instructions are organized in this plugin. It reflects an audit of the repository and the intended **artifact per folder** (not every folder is a `SKILL.md`).

**External reference:** Claude Code skills use `SKILL.md` with YAML frontmatter; plugins discover `skills/**/SKILL.md` and `agents/*.md` automatically. See [Claude Code: Extend with skills](https://code.claude.com/docs/en/skills) and [Plugins reference](https://code.claude.com/docs/en/plugins-reference).

## Directory map

| Path | Role | Format |
|------|------|--------|
| `skills/` | Executable capabilities; primary plugin surface | `*/SKILL.md` + optional `references/` (default enrichment: `references/deep-dive.md` — see [`skill-conventions.md`](../skills/_core/references/skill-conventions.md)), `routines/`, nested sub-skills. **Meta / helpers** (not regenerated by `apply_detailed_skills.py`): `skills/skill-creator/`, `skills/workflow-builder/`, `skills/shortcuts/*` (composite entry points). |
| `agents/` | Subagent / orchestration specs (when to use which skills) | YAML frontmatter + short body pointing to skills and guides |
| `prompts/` | Injectable fragments: preamble, personas, tones, policies, chains. **Tones, personas, and policies are org-customizable templates** — PMs maintain them so all agents communicate consistently. | Plain markdown with `<!-- [CUSTOMIZE] -->` markers; loaded by `!cat` from skills; config overrides via `jstack.config.json` keys (`tones.*`, `personas.*`, `policies.*`) |
| `templates/` | Human or machine output scaffolds (reports, SOP, tickets) | Placeholder markdown/JSON; not skills |
| `examples/` | Golden **input** / **output** pairs for skills and evals | Plain markdown; documents expected shapes |
| `dashboard/` | Next.js app (local UI) | No plugin skills; operator docs only (`README.md`) |
| `telemetry/` | Optional metrics pipeline (Zod, TS) | `README.md` for operators; implementation is code |
| `themes/` | UI color theme for dashboard | `jstack.json` + `README.md` for editors |

## Audit summary (high level)

- **`skills/`**: Many `SKILL.md` files follow a shared pattern (preamble, Instructions, Failure modes, `$ARGUMENTS`). See [`skills/_core/references/skill-conventions.md`](../skills/_core/references/skill-conventions.md) and [`markdown-authoring-guide.md`](../skills/_core/references/markdown-authoring-guide.md).
- **`agents/`**: There are **18** `agents/*.md` files (role routers plus orchestration-heavy specs). Preferred depth for maintainers: each agent should tie claims to repo evidence—**Specialty** (what a generic assistant would miss), **Configuration read order and unset behavior** (which `jstack.config.json` / `prompts/` knobs apply and safe degradation when unset), **Evidence chain (internal)** (`skills/**`, [`config-schema.md`](../skills/_core/references/config-schema.md), [`domain-map.md`](../skills/workflow-builder/references/domain-map.md)), and optional **External reference** (short URL + takeaway; no pasted excerpts). Cross-agent config summaries live in [`docs/agents-config-matrix.md`](agents-config-matrix.md). Rich agents (e.g. recon, chain orchestrator, workflow executor) spell out contracts, failure modes, and handoffs; thinner files remain **ordered skill routers** with explicit guardrails—both shapes are intentional. After editing agents—or renaming any `name: jstack-*` skill—run **`bun run agents-check`** from `jstack.core` ([`scripts/agents-check.ts`](../scripts/agents-check.ts) validates frontmatter and every `jstack:<suffix>` token in each agent file). Monorepo CI runs this when `jstack.core/agents/**` or the checker script changes; periodic runs before releases catch drift early.
- **`examples/`**: **46** files were `(stub)` placeholders; they are replaced with **realistic** input/output content so tools and evals have fixtures.
- **`dashboard/`**: No markdown; **`dashboard/README.md`** added for how to run and what it is for.
- **`themes/`**: JSON only; **`themes/README.md`** added.
- **`telemetry/`**: README extended beyond the drain-script note; schema and privacy called out.

## Conventions (quick)

1. **Preamble first**: Skills that integrate team context should reference `${CLAUDE_PLUGIN_ROOT}/prompts/setup/preamble.md` via `!cat` as in existing skills.
2. **No hardcoded secrets or team IDs** in any markdown; use `jstack.config.json` and `skill_defaults`.
3. **Chaining**: Use HTML `<!-- chains-to: jstack:... -->` and [`chaining-guide.md`](../skills/_core/references/chaining-guide.md) where a skill hands off to another.
4. **User interaction**: Prefer patterns in [`question-patterns.md`](../skills/_core/references/question-patterns.md); document optional args and when to ask in `SKILL.md` or an agent spec.
5. **Customizable templates**: Tones, personas, and policies use `<!-- [CUSTOMIZE] -->` markers where PMs should insert org-specific content. Each file includes a **Config hook** section showing the `jstack.config.json` keys that override defaults at runtime.

6. **Skill deep-dive files**: `skills/<name>/references/deep-dive.md` holds process, practices, and examples. `SKILL.md` should `!cat` it in Procedure step 1 when present (pilot list is wired in `scripts/apply_detailed_skills.py`).

7. **External PM skill libraries (link + license)**: See [`docs/SKILL_SOURCES.md`](SKILL_SOURCES.md) for curated links to [Product-Manager-Skills](https://github.com/deanpeters/Product-Manager-Skills) and [ar-claude-skills `product-team`](https://github.com/natea/ar-claude-skills/tree/main/product-team), mapping to jstack skills, and license notes. Prefer original jstack text or **link-out**; do not copy CC BY-NC-SA source verbatim into this repo without a legal decision.

8. **Composite shortcuts**: `prompts/shortcuts/composites.md` names recipes (`jstack:…` aliases) that combine persona/tone with external skills (`gstack`, `superpowers`). See also `skills/knowledge/shortcuts/`.
